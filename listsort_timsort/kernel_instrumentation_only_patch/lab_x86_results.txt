========================================
  Analysis Summary
========================================

XFS list_sort() usage characteristics:
1. Very high call frequency during file create/delete (13k+ calls per 30s)
2. Very small lists: avg 5 elements per call
3. High pre-sorting: 68-80% of calls have already-sorted input (1 run)
4. When not sorted: avg runs ratio ~28%, meaning moderate ordering
5. Ascending runs dominate (84-85% of all runs are ascending)
6. Single caller: all calls come from the same XFS code path

Conclusion:
- XFS heavily uses list_sort() but with very small lists (avg 5 elements)
- Natural run detection overhead (O(n) scan) on lists of 5 elements
  exceeds any possible benefit from fewer merge passes
- Run detection is most beneficial for larger lists (n > 100)
- For XFS's use case, the existing list_sort is already near-optimal
